KEY FINDINGS
─────────────────────────────────────────────────────────────────────────────

THREE BOTTLENECKS IDENTIFIED (Ranked by Impact):

1. REFLECTION OVERHEAD (25% of total time = ~2-3 hours)
   ─────────────────────────────────────────────────────
   Location: EstatisticaItemProcessor.getFieldValue()
   Problem:  Method.getMethod() + invoke() per column access
   Impact:   3,100 calls per chunk × 0.01-0.02s = 31-62s per chunk
   Root:     Java reflection overhead (10-20ms per dynamic method call)
   Fix:      Replace with MethodHandle caching (2-3x faster)
   Risk:     LOW (isolated refactoring)
   Effort:   4-6 hours development + testing
   
2. NETWORK LATENCY (30-40% of total time = ~3-4 hours)
   ──────────────────────────────────────────────────────
   Location: JdbcBatchItemWriter (SQL INSERT/UPSERT)
   Problem:  249 separate SQL executions, each ~50s (AWS↔Locaweb latency)
   Impact:   249 round-trips × 50s = 12,450 seconds = 3.5 hours
   Root:     Network latency (100-300ms per SQL execute) + commit overhead
   Fix:      Stored Procedure (1 SQL call) OR Async ItemWriter (pipelining)
   Risk:     MEDIUM (async) / LOW (stored proc)
   Effort:   4-12 hours development + testing
   
3. CHUNK OVERHEAD (10-15% of total time = ~98 minutes)
   ──────────────────────────────────────────────────
   Location: Spring Batch chunk management
   Problem:  249 chunks = 249 context switches + commits
   Impact:   Reduced throughput, per-chunk startup cost
   Fix:      Increase chunk-size from 50 to 500 rows
   Risk:     VERY LOW (config change)
   Effort:   30 minutes
